--- title: Accessibility description: Simple accessibility pointers for creating inclusive web experiences layout: layouts/_guide:text js: - contrast --- {% markdown %} Building interfaces and experiences that are accessible to people with a diverse range of hearing, movement, sight, and cognitive abilities is a core concern of Calcite Web. To that end all of our components are designed to be WCAG 2.0 AA compliant. As a designer or a developer using Calcite Web, there are several things you can do to ensure that the experiences you build with Calcite Web will work for everybody.

1. Ensure Sufficient Contrast

If text is set using two colors that don't have enough contrast, users who have impaired vision may not be able to read or understand the interface. To pass the WCAG 2.0 AA requirement guidelines, you should aim to have a color contrast of at least 4.5:1. {% endmarkdown %} {% include "guides/_color-finder.html" %} {% markdown %}

2. Skip to Content

A skip to content link should be used at the beginning of a page to allow users to bypass site navigation and immediately go to the primary content of the page. Calcite Web provides a helper class for links like this named `skip-to-content`. Simply create an anchor, and make the `href` point to an element located where your content starts. For example, use the following element as the first in your document: ```html Skip To Content ``` Then, wrapping the main content of the page, use an element with a `tabindex` of `0` and an `id` of whatever you linked to with the skip to content element: ```html

3. Use Semantic Markup

An important step towards building accessible experiences is using semantic markup. Many accessibility errors are simply the result of using wrong element for the job. For example, many developers will use a link instead of a button:
Incorrect: ```html Click Me! ```
Correct: ```html ```
If you need to style a button to look like a link, [use the `btn-link` class](../documentation/components/#button-links). For a more in-depth overview of how to write html in an accessible way, check out [Writing HTML with accessibility in mind](https://medium.com/alistapart/writing-html-with-accessibility-in-mind-a62026493412). Make sure you're familiar with HTML5 tags like `aside`, `figure`, `article`, `footer`, `nav`, `header`, and the more modern `input` types like `search`. MDN has a fairly good intro on the [WAI-ARIA basics ](https://developer.mozilla.org/en-US/docs/Learn/Accessibility/WAI-ARIA_basics) page.

4. Provide Focus States

It's very important that as a web developer you avoid hiding the browser's focus state. Turning off the default browser focus outline will make it very hard for people using keyboard navigation to know where they are in the document. If you do want to remove the default `focus` style from the browser, make sure you provide a custom style with sufficient perceptible change that is easy for users to see and interpret.
Incorrect: ```css .my-cool-element:focus { outline: none; } ```
Correct: ```css /* leave the browser default */ .my-cool-element:focus {} /* or create a custom style */ .my-cool-element:focus { outline: none; border-color: $blue; box-shadow: 0 0 2px 0 rgba(0,121,193,1); } ```

5. Use Correct Aria Roles (when needed)

Aria roles, especially the Landmark roles, are very important for users navigating sites using assistive technologies like screen readers. They help users understand the different areas of your experience and infer meaning about what they do. To that end, it's important that we use the correct html tags and ARIA roles. Many times, using the correct HTML5 tag is enough to infer meaning, but sometimes you will need to add an additional `role` attribute.

For a full list of aria roles, check out [Using ARIA](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/ARIA_Techniques) on MDN.

6. Use an Auditing Tool

There are many other small things that need to be done to make sure your site is accessible like `alt` tags for images, proper labels, avoiding repetitive links, descriptive link text, and more. The best way to make sure all of these smaller things get done correctly is to use an auditing tool. There are several great solutions for checking accessibility, many of which are now built into the development tools you already use. If you're a Chrome user, you can use Lighthouse (from the "Audits" tab) or install something more robust like the [Axe chrome extension](https://chrome.google.com/webstore/detail/axe/lhdoppojpmngadmnindnejefpokejbdd). Axe also provides an [extension for Firefox](https://addons.mozilla.org/en-US/firefox/addon/axe-devtools/). You could also use continuous integration tools in conjunction with something like [a11y](https://www.npmjs.com/package/a11y) to check your work every time you push. By making accessibility auditing part of your everyday workflow, you'll help to ensure the experiences you build will work for _everybody_. {% endmarkdown %}
